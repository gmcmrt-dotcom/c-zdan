# Hard rules (Wallet)

These rules are invariant. They are enforced by the service layer
(`apps/api/src/services/*` and `apps/api/src/middleware/*`) and audited via
`merchant_api_calls`, `audit_log`, and `merchant_settlement_log`.

1. **Idempotency** ‚Äî every merchant call requires `merchant_ref` (header `x-merchant-ref` for the public API, body field for callbacks). A second call with the same ref returns the cached response. The duplicate-mismatch case returns `DUPLICATE_REF` (HTTP 409).
2. **Signature** ‚Äî `HMAC-SHA256(signing_secret, timestamp + ":" + raw_body)`. In the commerce parent/child model, **the signing secret belongs to the parent**; child API calls carry the child `api_key` but verify against the parent `signing_secret`. Implementation: `apps/api/src/lib/merchant-hmac.ts`.
3. **Timestamp** ‚Äî ¬±5 minute window (`STALE_TIMESTAMP` on rejection). The middleware reads `x-merchant-timestamp` (unix seconds).
4. **Audit** ‚Äî every merchant call lands in `merchant_api_calls` (endpoint, method, ip, request_body, response_body, status_code, error_code, latency_ms). Wallet-side ledger writes go to `transactions` + `provider_ledger`; balance movement is logged in `merchant_settlement_log` (commerce) or `merchant_cash_pool_log` (finance).
5. **PII masking** ‚Äî the frontend uses `mask*()` helpers from `apps/web/src/lib/mask.ts`; full visibility requires `members.pii:view_full` permission (admin-only by default in the seeded `bo_permissions`).
6. **Service-layer scoping** ‚Äî there is **no Postgres RLS**. Every read goes through a service helper that pins the query to the caller's identity:
   - Member routes: `requireAuth` middleware ‚Üí `user(req).id` ‚Üí `makeUserDb(userId)` / direct scoped query.
   - Staff routes: `requireStaff` + `loadUserPerms` ‚Üí permission check via `can(perm)`.
   - Merchant BO routes: `requireMerchant` ‚Üí `req.merchantUserCtx.merchantId` filter on every query.
7. **No merchant name shown to the member** ‚Äî in flows A/B/C/D the member-facing UI never renders a merchant name, `description` (which may contain a merchant name), or `merchant_ref`. The member sees `txTypeLabel(tx.type)` + date + amount + `<TxIdBadge publicNo={tx.public_no} />`. Admin and merchant BO surfaces display the merchant name normally.
8. **No commission charged to the member** ‚Äî `transactions.amount` is the gross the member sees; `transactions.fee` is platform commission and never deducted from the member.
8.1. **Merchant accounting is net** ‚Äî per transaction: `transactions.amount` is gross, `transactions.fee` is commission. Flow A: `settlement += amount - fee`. Flow B: `settlement -= amount + fee`. Flow C: `cash_pool += amount - fee`. Flow D: `cash_pool -= amount - fee`.
9. **Reserve pattern** ‚Äî when a payment code is generated, both the spend points and the wallet balance are reserved (`accounts.reserved_balance`). They are consumed on Flow A success and released on cancel/expire.
10. **Tier snapshot** ‚Äî `payment_codes.reserved_at_tier_id` and `reserved_at_turnover` are written at code generation time (fairness ‚Äî a member can't grind to the next tier between code creation and consume).
11. **Merchant cash-pool / credit-limit guard** ‚Äî on `credit_member` (Flow B only), the service does `SELECT balance, credit_limit FROM merchants WHERE id=$1 FOR UPDATE`, computes max Flow B debit capacity as `balance + credit_limit` (overdraft ceiling when the settlement book is insufficient ‚Äî **not** a general withdrawable balance), and rejects with `INSUFFICIENT_MERCHANT_BALANCE` if `amount + commission > available`. A DB CHECK constraint `balance >= -credit_limit` is the last line of defence.
12. **`balance` ‚âÝ `cash_pool`** ‚Äî `merchants.balance` is the platform's settlement-book position for that merchant (signed; positive = prepaid, negative = owes us). `merchants.cash_pool` is the merchant's own bank-side liquidity used by Flow D routing for finance merchants. They never share writes.
13. **Merchant BO isolation** ‚Äî merchant users only see their own merchant's rows. Cross-merchant data and member PII never reach the merchant BO. Implementation lives in `requireMerchant` (`apps/api/src/middleware/auth.ts`) and `merchant.service.ts` / `merchant-public.routes.ts`.
14. **Three-layer transaction identity** ‚Äî
    - **`public_no`** (`<PREFIX>-YYYYMMDD-NNNNNN`) ‚Äî human-readable end-to-end ID. Stored on `transactions`, `topup_sessions`, `withdraw_sessions`. Generated in the service layer via `apps/api/src/lib/public-no.ts::allocPublicNo` against the `public_no_counters` table (single `INSERT ‚Ä¶ ON CONFLICT DO UPDATE RETURNING` per allocation). For topup/withdraw, the session's `public_no` is inherited by the transaction (member and merchant see the same ID start to finish). _(Historic note: an early Supabase version of this project used a `_set_tx_public_no` PostgreSQL trigger + `metadata`-copy triggers. The Express + Drizzle port replaced those with explicit service-layer code; no triggers exist in the current schema.)_
    - **`merchant_ref`** ‚Äî merchant-side transaction ID, the idempotency key. Written explicitly by the service layer when the transaction row is inserted. Required for flows B/C/D, optional for A.
    - **`external_tx_id`** ‚Äî third-party (bank/card/crypto) reference. Written explicitly by the service layer (no trigger). Used in flows C/D.
    - **Prefix map:** `T` topup, `W` withdraw, `P` spend, `C` merchant_credit, `B` bonus, `R` referral_bonus, `A` affiliate_payout, `AC` affiliate_commission, `PS` profit_share, `X` unknown (see `tx_type_to_prefix(text)`).
    - **Adding a new `tx_type`:** update `tx_type_to_prefix` *and* the `txType` enum in `apps/api/src/db/schema/_enums.ts`.
    - **API response:** the public merchant endpoints (`/merchant-api/charge`, `/merchant-api/credit`, `/merchant-api/topup-callback`, `/merchant-api/withdraw-callback`) return `wallet_tx_no` (= `public_no`), `merchant_ref`, and `external_tx_id` in the JSON envelope. Backwards-compatible.
    - **Frontend:** always render with `<TxIdBadge publicNo={...} />` (click-to-copy, monospace). i18n key `common.txId.label`.
15. **Commerce parent/child** ‚Äî A commerce parent merchant is an integration / group account; the Flow A/B accounting writes against the **child** `merchants.id`, not the parent. Each child has its own `api_key` + uses the parent `signing_secret`. The parent `balance` is an operational aggregate; the source of truth is `merchant_settlement_log.merchant_id = <child_id>`.
16. **Commerce merchant cashout** ‚Äî A commerce merchant can withdraw a positive settlement balance to crypto. Parent-level aggregate cashout is not allowed; the parent picks a child to act on behalf of. The request reserves `amount + fee` into `merchants.cashout_reserved_amount`. On success callback, `merchants.balance` decreases by the same total and `merchant_settlement_log.reason = 'merchant_cashout_paid'` is written. On failure, the reserve is released and the balance stays put. The starter must be the merchant owner or a user with the `merchant_cashout:create` sensitive permission.
17. **Session security state changes invalidate all tokens** ‚Äî `revokeAllForUser(userId, trx?)` MUST be called inside the same transaction as any of: password change, email change (member self-service OR staff-approved PCR), MFA enroll, MFA unenroll, account freeze, or permission elevation that raises the user's privilege level. Issued refresh tokens are killed; access-token denial relies on the access TTL (15 min default) plus the JTI denylist (`apps/api/src/auth/jti-denylist.ts`) for emergency revocation.
18. **Audit log writes are transactional** ‚Äî `writeAudit({ trx })` (`apps/api/src/services/admin/audit.ts`) accepts an optional Drizzle tx handle so an audit-row failure rolls back the mutation it describes. State-changing routes pass `actorId`, `ip` (`clientIp(req)`), `userAgent` (`req.get("user-agent")`), and the relevant `before`/`after` snapshots. PII is masked at the writer via `redactForStorage`.
19. **PCR (profile-change request) integrity** ‚Äî member-side PCR routes accept ONLY `first_name` / `last_name`. Email and phone changes go through the dedicated OTP flows (`/api/auth/profile-change-otp`). Approval AND rejection write an `audit_log` row inside the apply transaction with `before.old_value` snapshot + `after.new_value` (masked). Email approvals additionally clear `users.email_verified_at` and call `revokeAllForUser` per rule 17.
20. **Audit + chat retention is FOREVER** ‚Äî `audit_log`, `merchant_api_calls`, `merchant_settlement_log`, `merchant_cash_pool_log`, `chat_threads`, `chat_messages`, `chat_profile_change_requests`, and `ai_cost_log` are never deleted by the app. They MAY be partitioned by month (Postgres native) once `merchant_api_calls` crosses ~100M rows. There is no scheduled purge cron for these tables. (Token tables ‚Äî `auth_refresh_tokens`, `password_reset_tokens`, `email_verification_tokens`, `profile_change_otps` ‚Äî DO have a purge cron because they hold dead bearers; that's `purge_expired_auth_tokens` in `apps/api/src/workers/scheduler.ts`.)
21. **i18n `escapeValue: false`** ‚Äî i18next is configured with `escapeValue: false` because every translation key is static JSON authored by us, AND every render is via React JSX which escapes the value during render. If you ever introduce a translation that receives user-controlled content, switch to `escapeValue: true` for THAT key (or wrap the call) ‚Äî do not switch the global default without auditing every translation that uses intentional HTML markup.
22. **Local-only storage / no APM / offline geo** ‚Äî chat attachments and uploads live on local disk at `STORAGE_LOCAL_DIR` (the wallet-api host). No S3 / R2 dependency. APM is intentionally **not** wired ‚Äî Pino structured logs are the source of truth; the deploy log forwarder is responsible for routing errors to Telegram / email. AI spend is tracked in `ai_cost_log` (rule 20) and gets a soft alert via the hourly `ai_budget_alert` cron at 80% of `AI_DAILY_BUDGET_USD` (default 50). No upload virus scanner is bundled; deploy-side ClamAV is the documented approach. Geo enrichment on `user_login_ips` runs via **`geoip-lite` (offline MaxMind GeoLite2)** ‚Äî zero outbound network calls, no PII goes to a 3rd party. Refresh the bundled DB quarterly via `npx geoip-lite-update` in CI. `country` / `country_code` are reliable; `city` / `region` are best-effort (often null for non-US blocks).
23. **MFA backup codes (K3)** ‚Äî `mfaVerifyEnrollment` returns 8 one-time recovery codes ONCE. They are stored only as sha256 hashes in `user_mfa_backup_codes`. The challenge endpoint accepts a backup code in place of a TOTP code (format `XXXXX-XXXXX`, 10 alphanumeric chars excluding I/O/0/1) and marks it consumed atomically. Regeneration requires aal2 and invalidates the entire old set.
